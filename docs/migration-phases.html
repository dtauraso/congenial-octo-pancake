wirefold / migration-phases

Two-store + substrate-r → RF-only editor; Go is the only canonical runtime

Phase 1 — POC (verified)

Proved that a node body and pulse animation can live entirely inside RF custom node/edge components with no substrate-r mounting layer. See docs/phase-1-verification.md for the full trace.
Conclusion (stronger than the original framing)
The POC was framed as "substrate runs inside RF." The actual result is stronger: the editor can be RF-only. No TS substrate library, no shared substrate/ module, and no TS runtime survive. Go is the canonical runtime; the TS editor is a preview/simulation only.
Done. Phase 2 is the active work.

Phase 2 — Delete substrate-r, rewrite kinds RF-native

Delete tools/topology-vscode/src/webview/substrate-r/ in its entirety. Rewrite every node kind from scratch as self-contained RF custom node components that render the topology statically. Wire the Run button to invoke the Go runtime; pulse animation from Go trace events lands in Phase 4. Old substrate-r files may be read once as reference; new code is RF-native — no ports, no adapters.
Implementation contract
Per-kind RF nodes render the topology and accept user edits via RF primitives (node.data, setNodes, setEdges, useReactFlow). They do not simulate. Run invokes the Go runtime; Go is the only runtime. No shared substrate/ module, no slot phase machine in TS, no TS-side enforcement of substrate semantics — parity with Go comes from Go being the reference.
Phase 2 → Phase 4 gap (intentional)
After Phase 2 and before Phase 4, Run invokes the Go runtime but the editor does not yet render pulses — the trace-stream consumer lands in Phase 4. The editor is fully usable for editing during the gap; live visualization is the Phase 4 deliverable.
Steps
  • Start with InputNode: create rf/nodes/InputNode.tsx — RF custom node that renders the kind's static visual (ports, label, init-queue display) from node.data. No simulation logic, no RAF.
  • Create rf/edges/SubstrateEdge.tsx: RF custom edge that renders the wire's static visual (path, label, kind styling). No pulse animation yet — Phase 4 adds that on trace events.
  • For each remaining kind (Relay, Join, ReadGate, ReadLatch, Partition, Edge, Inhibitor, ChainInhibitor, EdgeInhibitor, InhibitRightGate, SyncGate, StreakDetector, StreakBreakDetector, TransferInhibitor): create one file under rf/nodes/<KindName>Node.tsx with port list and static visual. No bodies, no simulation.
  • Register all kinds in rf/app/AppView.tsx nodeTypes/edgeTypes maps.
  • Wire the Run button to invoke the Go runtime (subprocess or IPC). The editor does not consume Go output yet; that lands in Phase 4. Run goes from faded to live in this phase.
  • Delete tools/topology-vscode/src/webview/substrate-r/.
  • Confirm no imports of substrate-r remain: grep -r "substrate-r" --include="*.ts" --include="*.tsx" tools/topology-vscode/src/
  • Run npm run build and verify it passes.

Phase 3 — Single state source (RF only)

The editor's only state is RF state. Delete the second Zustand store, specToFlow, mutateBoth, and the parallel undo/redo stacks. RF node/edge data fields are the sole editor state.
Steps
  • Define typed NodeData and EdgeData schemas that carry all fields currently split across the Zustand spec store and RF node data.
  • Replace spec-to-flow.ts projection with direct setNodes/setEdges calls using the typed data fields.
  • Wrap setNodes/setEdges in validation dispatchers (rf/nodes/dispatchers.ts); these are the only mutator API — enforce via module boundary.
  • Delete specToFlow, mutateBoth, and the Zustand spec slice.
  • Delete parallel undo/redo stacks; verify undo/redo works via RF's own snapshot mechanism or a thin wrapper.
  • Update SaveLifecycle to read from RF state (useNodes/useEdges) directly.
  • Grep for any remaining imports of the deleted Zustand store and resolve.

Risks
Dispatcher bypass
Any code path calling raw setNodes/setEdges directly bypasses validation. Enforce module boundary. Grep for direct calls as audit step before deleting old store.
Undo granularity

Done when: no Zustand store outside RF's own internals; build passes; save/load round-trip preserves all node fields; undo/redo works for add/delete/rename.

Phase 4 — Codegen / Go handshake

Simplify topology.json to RF's native shape; topogen reads RF state. Run button calls the Go runtime; the editor visualizes the trace stream Go emits. Go is the reference — TS simulation is display-only and may drift.
The pump is a translator, not a substrate
RF does not read Go output directly. A thin TS module (rf/pump.ts or similar) listens for Go trace events and translates each one into an RF state update: edge.data.pulse = {value, simTime}, node.data.lastFire = simTime, etc. RF custom nodes/edges read that state and animate. The pump must contain zero substrate logic — no slot phase tracking, no backpressure, no fire scheduling. If the pump grows a state machine, it is becoming substrate-r v2; stop and route that logic into Go instead.
Steps
  • Update SaveLifecycle to serialise RF NodeData/EdgeData to the topology.json schema; topology.json remains as a snapshot artifact for git-diffability.
  • Update cmd/topogen/main.go input parser if RF-state schema diverges from current topology.json schema; add a schema version field and assert it in topogen.
  • Run go build ./... to confirm generated Wiring_gen.go compiles.
  • Wire the Run button to invoke the Go runtime (IPC or subprocess); receive the trace stream (Go events).
  • Drive pulse animations from Go trace events: each event carries node id, slot index, and simTime; RF custom nodes/edges animate on receipt. This is the only source of pulse animation — there is no TS fallback.

Risks
Schema drift
RF NodeData fields diverging from topology.json schema causes topogen to silently produce wrong wiring. Add schema version field; assert in topogen before parsing.
IPC latency
Go trace events arrive asynchronously; RF animation must buffer and replay at correct simTime offsets, not wall-clock time. Anchor each pulse at its emit simTime.
Go parity
TS preview may show behavior that differs from Go. Go wins — do not add TS-side enforcement of substrate semantics. If parity matters, add a Go test, not a TS check.
Done when: Run button is live; pressing it calls the Go runtime; the editor shows pulses driven by Go trace events; go build ./... and npm run build both pass.
