Two-store + parallel substrate-r → RF-only with substrate logic in custom nodes/edges

Phase 1 — POC: one kind end-to-end in RF

Prove that a substrate node body and pulse animation can live entirely inside RF custom node/edge components, with no separate substrate-r mounting layer.
Audit finding
The substrate-r audit found all node kinds port-ready — no logic rewrites needed. InputQueueEditor is the only "port-with-light-refactor" item (overlay positioning assumes a separate substrate canvas coordinate space; adjust to RF node-local coords).
Steps
  • Pick InputNode as the test case.
  • Create rf/nodes/InputNode.tsx: RF custom node that owns RAF loop, input slot refs, and the Input kind body inline.
  • Create rf/edges/SubstrateEdge.tsx: RF custom edge that reads the RF-internal path geometry and drives a pulse SVG circle along it.
  • Wire a 2-node toy topology (InputNode → SubstrateEdge → ConsumerNode) directly in RF — no SubstrateProvider, no separate registry.
  • Verify pulse fires on input delivery and animation duration matches today's substrate-r Wire.tsx output.
  • Confirm slot backpressure still works: a second load() while in-flight is a no-op.
Files touched
rf/nodes/InputNode.tsx
New: RF custom node with embedded Input body (RAF, slot refs, queue drain)
rf/edges/SubstrateEdge.tsx
New: RF custom edge with SVG pulse animation along the RF-computed path
rf/app/AppView.tsx
Register new node/edge types in the RF nodeTypes/edgeTypes maps for the POC
substrate-r/Wire.tsx
Read-only reference for animation timing constants and slot-fill contract
Risks
Path geometry access
RF custom edges receive the SVG path string but not the raw point array. Pulse position must be computed via SVGPathElement.getPointAtLength; verify this is available in the webview sandbox before committing to the approach.
Ref threading
Without SubstrateProvider, wire.fill() must reach the dest node ref via a shared module-level map. Establish the map boundary clearly to avoid a de-facto global registry that recreates what is being removed.
Done when: a 2-node RF-only topology runs, InputNode fires and drains its queue, the pulse animates along the edge, and the ConsumerNode slot fills — all without importing anything from substrate-r/.

Phase 2 — Port remaining kinds

Migrate every node kind out of substrate-r/ into self-contained RF custom node files, then delete the substrate-r/ directory.
Audit finding
The substrate-r audit found all kinds port-ready — no logic rewrites needed before porting. InputQueueEditor requires light refactoring: its overlay positioning assumes the substrate canvas coordinate space; update to RF node-local coordinates during the port.
Steps
  • For each kind in node-kinds.tsx (Input, Relay, Join, ReadGate, ReadLatch, Partition, Edge, Inhibitor, ChainInhibitor, EdgeInhibitor, InhibitRightGate, SyncGate, StreakDetector, StreakBreakDetector, TransferInhibitor): create one file under rf/nodes/<KindName>Node.tsx.

  • Update rf/app/AppView.tsx nodeTypes map to point to the new files.
  • Delete tools/topology-vscode/src/webview/substrate-r/.
  • Confirm npm run build passes with no imports of substrate-r.

Phase 3 — State registry for fold survival

Preserve per-node runtime state (slot phases, in-flight values) across React fold collapse/expand cycles by moving it from component-local refs into a module-level registry.
Steps
  • Create rf/nodes/runtime-registry.ts: a module-level Map<nodeId, RuntimeState> where RuntimeState holds slot phases and any in-flight wire values.
  • On mount, each RF custom node reads its entry from the registry (if present) and restores local refs.
  • On unmount, each node writes current slot state back to the registry.
  • Wire phase (in-flight) needs the same treatment on the edge side: rf/edges/SubstrateEdge.tsx saves/restores animation progress on unmount/mount.
  • Test: start a simulation run, collapse the fold containing an in-flight wire, expand immediately, confirm the pulse resumes rather than drops.
Files touched
rf/nodes/runtime-registry.ts
New: module-level Map holding RuntimeState per node id
rf/nodes/<Kind>Node.tsx
Each kind: add mount-read and unmount-write hooks against the registry
rf/edges/SubstrateEdge.tsx
Save/restore animation progress (RAF t, in-flight value) on unmount/mount
Risks
Fold + running sim
A fold collapse while the RAF loop is mid-tick can leave the registry write racing the animation frame. Halt the node's RAF before unmount write; resume after mount read.
Stale registry entries
If a node is deleted (not folded), its registry entry must be evicted. Hook deletion through the same path that removes the RF node to avoid phantom entries accumulating.
Done when: collapsing a fold group mid-simulation and immediately expanding it shows slot state intact (filled slots remain filled, in-flight pulse resumes) with no console errors.

Phase 4 — Collapse the second Zustand store

Move the topology spec out of the parallel Zustand store and into RF's own node/edge data fields, leaving a single source of truth.
Steps
  • Define typed schemas for RF node data (NodeData) and edge data (EdgeData) that carry all fields currently in the Zustand spec.
  • Replace spec-to-flow.ts projection with direct RF setNodes/setEdges calls using the typed data fields.
  • Wrap setNodes and setEdges in validation dispatchers that run the existing spec validators before committing; these are the new mutator API.
  • Delete specToFlow, mutateBoth, and the viewer history undo/redo stacks that were backed by Zustand spec snapshots.
  • Verify undo/redo still works via RF's own snapshot mechanism or a thin wrapper around it.
  • Verify save: SaveLifecycle reads from RF state directly and serialises to topology.json.

Remove spec slice; keep only non-topology fields (runStatus, nextTxnId) if still needed
state/spec/mutators.ts
Delete; logic moves into validation dispatchers

Delete; no longer projecting from a separate store
rf/nodes/dispatchers.ts
New: typed setNodes/setEdges wrappers with validation
webview/SaveLifecycle.tsx
Update to read from RF state (useNodes/useEdges) instead of Zustand spec
state/history.ts
Delete or replace with RF-native snapshot approach
Risks
Validation dispatcher bypass
Any code path calling RF setNodes/setEdges directly bypasses validation. Enforce by making raw setNodes/setEdges inaccessible outside dispatchers.ts (module boundary). Grep for direct calls in the audit step.
Undo stack regression
Zustand history.ts gave per-operation undo. RF's built-in history operates at the node-graph level. Verify undo granularity is acceptable before removing the old stacks.
Done when: build passes, no imports of state/store.ts for spec data, save/load round-trip preserves all node fields, and undo/redo works for add/delete/rename operations.

Phase 5 — Go codegen direct from RF state

Update topogen to read RF state at save time, eliminating or simplifying the topology.json intermediate format.
Steps
  • Decide serialization strategy (see Risks below): either keep topology.json as a snapshot of RF state, or emit Wiring_gen.go directly without an intermediate JSON file.
  • If keeping topology.json: update SaveLifecycle to serialise RF NodeData/EdgeData to the existing JSON schema; topogen remains unchanged.
  • If going direct: topogen reads from a new RF-state export (JSON-lines or inline call); delete topology.json from the save path; add a snapshot command for git-diffable artifacts.
  • Validate topogen output against the existing Wiring.go contract: same channel names, same node constructors.
  • Run go build ./... to confirm the generated file compiles.

Generated output; format unchanged, input path may differ
topology.json
May become a snapshot artifact rather than the live spec, or be retired entirely
Risks
Snapshot vs direct codegen
Keeping topology.json preserves git-diffability of topology changes and lets topogen remain a standalone CLI. Skipping it saves a round-trip but loses that audit trail. Recommended: keep topology.json as a snapshot; it is cheap to write and valuable for diffs.
Schema drift
If RF NodeData fields diverge from the current topology.json schema, topogen will silently produce wrong wiring. Add a schema version field and assert it in topogen before parsing.
Done when: saving the editor writes topology.json from RF state, topogen runs without error, go build ./... passes, and a round-trip (edit → save → run) produces correct Go output.
