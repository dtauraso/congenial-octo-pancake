Naming diagram nodes & edges — reduce navigation tax

Goal: names that tell the AI (and you) the relationships, so the substrate is navigable by structure instead of by string search.

A navigation tax is what happens when the AI can’t navigate by structure and has to triangulate by string fragments — pulling files it doesn’t need, reading dead code, asking questions the names should have answered. Every grep is wasted context and a step closer to a poisoned window. Clean naming is the single biggest lever on AI cost and output quality.

Why drift happens — one concept, five boundaries

In wirefold one concept (say “chain inhibitor stage 1’s output port”) crosses at least five files in different languages and conventions. Every boundary is a place a name can drift.

The cost of navigation tax

  • Tokens. Every grep pulls more file content than it needed, into the next prompt.
  • Wrong context. Grep returns dead code, comments, and stale references alongside the live one. The AI can’t tell which is which without reading more — more grep, more tokens.
  • Hallucination risk. When the live reference looks like the dead one, AI guesses. The guess looks plausible. The fix breaks the wrong site.
  • Compounding. Each prompt that grepped poisons the next prompt’s context. Twenty messages in, the conversation is half noise from earlier searches.

Naming rules

RULE 1 One canonical noun per concept, across every boundary. The same string (modulo unavoidable case rules of the host language) must appear in the Go struct field, the topogen registry key, the topology JSON handle, the editor port, the SVG group id, and the channel name. Case can flex (I1Out in Go, i1Out in JSON) but the identifier root is identical. AI can grep one string and find everything.
RULE 2 Channel/edge names encode both endpoints. House rule already in CLAUDE.md. i1Out_to_readGateI1In beats chan2. The name tells you the topology without opening the file. For SVG, the edge group id mirrors the channel name.
RULE 3 Ports are named for the concept they carry, not their direction. inputValue, i1Out, heldSeed tell the reader what flows. in, in2, out tell the reader nothing — they force the reader to chase the wiring to figure out what’s flowing. Direction is structural (in vs out is already in the type system); the concept is the part that needs a name.
RULE 4 Instance-specific when there are multiple instances. If a topology has two chain inhibitors, their port names must encode which inhibitor — i0Out, i1Out, not chainOut. The substrate model rule established 2026-05-18: node struct fields and port names are topology-instance-specific; the editor renames them when the topology changes.
RULE 5 No abbreviation unless the full form appears nowhere else. chainIn for chainInhibitorInput is fine if the file has no other chain-anything. chainIn next to chainInh next to chInh is the classic drift that produces navigation tax. Pick one and commit. Abbreviation is paid for by uniqueness, not by saving four characters.
RULE 6 SVG groups carry data-role and data-index. Per svg-style-guide.md: <g id="node-i1" data-role="inhibitor" data-index="1">. Edges: id="edge-i1Out_to_readGateI1In" data-source="i1" data-target="readGate". AI scrapes this and reasons structurally instead of by visual proximity. SVG becomes a queryable graph.
RULE 7 File name = concept name. If the concept is ChainInhibitor, the file is ChainInhibitor.go, ChainInhibitor.tsx (or the kind-body file references it by that exact noun). Avoid utils, helpers, common, misc — those are anti-files. Per the project’s concept-bounded carve-out for substrate-r/: one file per model concept.
RULE 8 The name has to survive the next rewrite. If you wouldn’t pick this name in a clean rewrite from spec, don’t pick it now. Names locked in early become load-bearing — every downstream reference encodes them. Pick the name the spec would use, not the name the code currently uses.

Concrete pass — converging the chain-inhibitor vocabulary

One bounded branch. Walk the five boundaries, converge each to a single canonical noun, push. Use AI to map and edit; you decide the names.

The pass in commands (rough sketch)

  1. Map: AI greps all spellings of the concept. Returns a table: file · line · current spelling · role.
  2. Decide: You pick the canonical name. Write it on a sticky note above the keyboard.
  3. Edit: One subagent per boundary, each with a tight prompt: "rename X to Y in this file, preserve everything else." Commit per boundary.
  4. Verify: Build the Go side, type-check the TS side, run the editor, exercise the animation.
  5. Grep check: Run rg "old-spelling-1|old-spelling-2|..." against the repo. Zero hits means the convergence is complete. Any hits = follow-up commit.

Pre-commit naming checklist

Run this against any branch that introduces, renames, or reshapes a node, port, or edge.

  • ☐ One canonical noun. The new name appears identically (modulo language case rules) in Go, topogen registry, JSON, editor, SVG.
  • ☐ Edge name encodes endpoints. source_to_target pattern; channel name in Wiring.go matches edge id in topology JSON matches id on the SVG <g>.
  • ☐ Port concept, not direction. Port names describe what flows, not in/out (the type carries direction).
  • ☐ Instance-specific. If there are multiple instances of the kind, each port name encodes its instance index or role.
  • ☐ No abbreviation drift. rg "abbreviated|abbreviated2|fullForm" — only the chosen form appears.
  • ☐ No utils/helpers/common files introduced or extended. New code lives in a concept file.
  • ☐ SVG groups tagged. Every <g> has id + data-role; numbered instances have data-index.
  • ☐ No Java-isms. No Handler, Manager, Controller, Service, Helper, Util suffix unless the substrate genuinely names that role.
  • ☐ Case style consistent. Same case at every boundary (camelCase end-to-end in this repo, including JSON).
  • ☐ Banned-vocab check passes. node tools/topology-vscode/scripts/check-substrate-vocab.mjs green.
  • ☐ Old spellings return zero grep hits. rg for every prior name; the rename is complete or there’s a follow-up commit on the same branch.
  • ☐ Rewrite-survival test. Ask yourself: "If I rewrote this from spec tomorrow, would I pick this name again?" If not, pick the rewrite-version now.

Why this list exists

Every item is a place AI defaults will drag the substrate toward training-data conventions. The list is the encoded form of your judgment so it doesn’t have to be re-argued each prompt. Same shape as MODEL.md and the banned-vocab check: encode the discipline, don’t rely on conversational pushback.

The compounding payoff: every naming pass that lands cleanly makes the next pass cheaper, because the AI has more clean examples to extrapolate from. Five or six passes in, the model produces good names by default and the checklist is mostly a formality. That’s the substrate cleaning itself — the opposite of the dirty-repo death spiral.
